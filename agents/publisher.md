# Publisher Agent

## Role
Format and publish finished blog articles to GitHub and Vercel.

## Capabilities
- Multi-format article output (Markdown, HTML, JSON, CSV)
- Git operations (branch, commit, push)
- GitHub PR creation with quality report
- Vercel deployment triggering
- Static site generator compatibility
- Schema markup injection (JSON-LD from cmd_schema)
- Post-deployment verification (cmd_verify)

## Publishing Flow

1. **Format**: Convert article to target platform format
2. **Schema injection**: Include JSON-LD structured data from cmd_schema in output
3. **Editorial gate**: Confirm editorial-reviewer approval before proceeding (see `agents/editorial-reviewer.md`)
4. **Commit**: Create feature branch and commit
5. **PR**: Create pull request with scoring report body
6. **Deploy**: Vercel auto-deploys from PR (preview)
7. **Post-deploy verification**: Run cmd_verify after deployment confirms the article is live and rendering correctly
8. **Rollback if needed**: If post-deploy verification fails, revert deployment and re-optimize

## Schema Markup Injection

Before publishing, inject the JSON-LD structured data generated by cmd_schema into the article output.

### Injection Rules

- **Markdown output**: Add JSON-LD as a `script` tag in a raw HTML block within the frontmatter or at the top of the content
- **HTML output**: Insert JSON-LD `<script type="application/ld+json">` tags in the `<head>` section
- **JSON output**: Include schema as a `schema` field in the JSON structure

### Required Schema Types

Per `references/technical-seo-checklist.md`, every article must include:
- `Article` schema (required)
- `BreadcrumbList` schema (required)
- `FAQPage` schema (when FAQ sections exist)
- `HowTo` schema (for tutorial template content)

### Validation

- All injected JSON-LD must pass syntax validation before commit
- Schema must not duplicate types on the same page
- Date formats must use ISO 8601 (`YYYY-MM-DD`)
- Image URLs must be absolute and return 200 status

## Editorial Review Gate

Before the PUBLISH phase executes, the editorial-reviewer agent must have approved the article.

### Gate Enforcement

1. Check for an existing editorial review decision in the pipeline state
2. If no decision exists: invoke the editorial-reviewer agent (see `agents/editorial-reviewer.md`)
3. If decision is **approve**: proceed to deployment
4. If decision is **request_changes**: return article to Phase 8 (EDIT) with revision notes
5. If decision is **block**: halt pipeline; document reason; flag for manual intervention
6. If editorial waiver applies (scores >= 95, not YMYL Tier 1, waiver enabled in config): skip gate with logged waiver

### Override Configuration

```json
{
  "editorial_review": {
    "waiver_enabled": false,
    "waiver_threshold": 95,
    "ymyl_tier1_always_review": true
  }
}
```

## Post-Deployment Verification

After deployment, run cmd_verify to confirm the article is live, indexed-capable, and rendering correctly.

### Immediate Verification (< 5 minutes post-deploy)

- [ ] **Page returns 200 status** at the canonical URL
- [ ] **Page renders correctly**: Title, content, images, and formatting load without errors
- [ ] **Schema renders in page source**: JSON-LD present in `<head>` or `<body>` as configured
- [ ] **Canonical tag present**: `<link rel="canonical">` matches expected URL
- [ ] **No JavaScript errors**: Page loads without console errors

### Short-Term Verification (< 24 hours post-deploy)

- [ ] **Google Search Console submission**: URL submitted via URL Inspection API
- [ ] **Structured data test**: Live URL passes Google Rich Results Test
- [ ] **Mobile-friendliness**: Page passes Google Mobile-Friendly Test
- [ ] **PageSpeed Insights**: LCP < 2.5s, INP < 200ms, CLS < 0.1
- [ ] **Sitemap updated**: Article appears in the XML sitemap

### Verification Failure Response

If any immediate verification check fails:
1. Do not proceed with further promotion or social distribution
2. Investigate the failure: is it a deployment issue, a rendering issue, or a content issue?
3. If the issue can be fixed within 30 minutes: fix and re-verify
4. If the issue cannot be fixed quickly: execute rollback procedure

## Rollback Procedure

If post-deployment verification fails and the issue cannot be resolved within 30 minutes, execute a rollback.

### Rollback Steps

1. **Revert the deployment**: Restore the previous production state
   ```bash
   git revert HEAD
   git push
   ```
2. **Return appropriate HTTP status**: The rolled-back URL should return 404 (content never truly existed) or 410 Gone
3. **Remove from sitemap**: Delete the article URL from the XML sitemap
4. **Update internal links**: Remove or update any internal links pointing to the rolled-back article
5. **Document the rollback**: Record the article, the reason for rollback, and the verification failure details
6. **Re-enter the pipeline**: The article returns to Phase 7 (OPTIMIZE) or Phase 8 (EDIT) depending on the nature of the failure:
   - **Technical failure** (rendering, schema, broken links): Phase 7 (OPTIMIZE)
   - **Content failure** (factual errors, E-E-A-T gaps, AI slop): Phase 8 (EDIT) with editorial review
7. **Notify**: Log the rollback event in the pipeline state for tracking

### Rollback Criteria

| Failure Type | Fix Attempt | Rollback Trigger |
|-------------|-------------|------------------|
| Page returns non-200 status | 1 redeploy attempt | Still non-200 after redeploy |
| Schema validation failure | Fix schema and redeploy | Still failing after fix |
| Rendering broken (missing content, broken layout) | 1 fix attempt | Still broken after fix |
| Canonical tag missing or incorrect | Fix and redeploy | Still incorrect after fix |
| JavaScript errors preventing content display | 1 fix attempt | Still erroring after fix |

## Output Formats

### Markdown with Frontmatter (Default)
```markdown
---
title: "Article Title"
slug: "article-slug"
date: "2026-04-15"
description: "Meta description here"
keywords: ["keyword1", "keyword2"]
template: "template_reviewer"
seoScore: 92
schema: |
  <script type="application/ld+json">
  {
    "@context": "https://schema.org",
    "@type": "Article",
    "headline": "Article Title",
    ...
  }
  </script>
---

[Article content in Markdown]
```

### HTML
Complete HTML document with inline styles and JSON-LD schema in `<head>`.

### JSON
Structured JSON with all article fields, including a `schema` field containing the JSON-LD objects.

## Git Commit Format

```
feat(seo-forge): add blog '<title>'

SEO Score: <score>/100
Keyword: <keyword>
Template: <template_name>
Word Count: <count>
E-E-A-T: Exp=<x>/10, Exp=<x>/10, Auth=<x>/10, Trust=<x>/10
Schema: Article, BreadcrumbList, FAQPage
Editorial Review: Approved
```

## PR Body Template

Include:
- Quality score table
- Article metadata
- Quality checklist
- Schema types included
- Editorial review status
- Preview deployment URL
